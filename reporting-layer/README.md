# Reporting Layer ‚Äî Phase 5 Metric Registry (Proof-of-Concept)

Date: 2026-04-09
Status: **Proof-of-concept.** Three example metric YAMLs + three matching source contract YAMLs. Not yet wired into CI. Not yet the primary source of truth (the existing `docs/specs/report-1-source-contract.md` and `docs/specs/report-2-source-contract.md` still serve that role during the transition).

## What this is

A structured, versionable, diffable metric registry for the Salesforce-reports-backed reporting layer. Applies the dbt MetricFlow / LookML / Cube.dev semantic-layer pattern to a substrate that is **Salesforce Reports** rather than a data warehouse.

## Metric file schema

Each metric YAML declares:

- `name`, `version` (semver), `display_name`, `description`
- `measure`: the aggregation ‚Äî `type` (sum/avg/count/etc), `field`, `currency_handling`, `aggregation_expression` (the literal SF Reports API string)
- `dimensions`: `time` (field + framing + grain) and `segment` (list of dimension fields)
- `filters`: list of `{field, operator, value}`
- `source`: `{report_id, developer_name, contract_file}`
- `bindings`: list of dashboard widgets that consume this metric
- `owner`, `reviewers`, `changelog`

## Source contract schema

Each contract YAML declares the **expected shape** of a Salesforce Report:

- `report_id`, `developer_name`, `display_label`, `report_type`, `format`
- `groupings_down`, `groupings_across`, `detail_columns`
- `filters`, `standard_date_filter`
- `aggregations` with explicit `currency_handling`
- `bucket_fields`, `calendar_framing`
- `normalization_notes`, `known_gotchas`
- `hash` (normalized SHA256 for drift detection)
- `bound_metrics` (back-reference)

## Drift detection (planned)

A scheduled CI job pulls each report's `describe` via the Analytics REST API, normalizes the `reportMetadata` (sorts keys, strips churn fields like `lastModifiedDate`, strips localized labels), hashes it, and compares to the `hash` field in the contract. Mismatch opens a PR with the diff as the body.

The existing `scripts/dashboard_state_dump.py` is a simpler precursor ‚Äî it checks semantic-level drift flags (no-convert, fiscal-grouping, etc.) but not the full normalized-hash diff. Phase 5 extends this with the hash-diff pattern.

## Migration plan (Phase 5)

1. Write metric YAMLs for every production metric (start with the 3 stubs here, expand as each metric is touched)
2. Write contract YAMLs for every source report
3. Generate `hash` fields via `python3 scripts/normalize_and_hash.py --all`
4. Wire the drift detection script into CI (GitHub Action or make target)
5. Retire `docs/specs/report-{1,2}-source-contract.md` once all metrics + contracts exist here
