## Hard rules applied to this dashboard

1. **Calendar year only.** No FY labels. Use `Q2 2026` (calendar), not `Q2 FY27`. Same hard rule as Report 1.
2. **ARR canonical for value aggregation.** Use `APTS_Opportunity_ARR__c` for pipeline value, not `Amount`. Per Sales Handbook V4 slides 24-25.
3. **No em-dashes** anywhere in widget labels. Use hyphens, periods, or rephrase.
4. **No gauges or donuts** in widget visuals. Bullet charts, ranked bars, sparklines only.
5. **Forecast accuracy uses Pipeline Inspection native, not in-house mirrors.** Per Andre's "turnkey standardization, layer of protection" instruction (2026-04-07). PI gives standardized metric definitions that don't drift; bespoke forecast metrics (the in-house Forecast_Intelligence CRMA dashboard, the dead `Opty_Weekly_Snapshot__c` table) are explicitly NOT canonical for this report.
6. **Process compliance is anchored to Sales Handbook V4 stage qualifiers.** Per slides 8-9, each stage has explicit exit gates that must be satisfied before a deal advances. Process compliance metrics are derived from these gates. Long-term: per-gate measurement requires Opportunity Plan / Sales Process Manager field instrumentation; short-term: the spec lists SF-native field-level proxies that are gradable today.

## Open questions

Resolved during the 2026-04-07 brainstorm:

- ~~Is Report 2 = 4 unique KPIs only or 9 (including Report 1's 5 bullets)?~~ **Resolved: 4 only.** Confirmed by Dashboard 2 having zero widgets for any of Report 1's 5 bullets.

Remaining:

1. **Process compliance: full handbook exit gates vs SF-native proxies.** The handbook (slides 8-9) lists ~30 exit gates across stages 1-6. Measuring each requires Opportunity Plan / Sales Process Manager instrumentation that may not exist as standard SF fields today. The spec uses 5 SF-native proxies (widgets 6-10) as a first pass. Long-term: Phase 3 should investigate the Sales Process Manager tool surface and expand process compliance to per-gate measurement. _Current assumption: 5 proxies are good enough for Report 2 v1; per-gate compliance is a Phase 3 enhancement._
2. **`Missing Quote Type` widget** (widget 2): the underlying `APTS_Primary_Quote_Type__c` field has zero active picklist values in the org (verified via sobject describe in Phase 1). The widget grades a field that nothing populates, so 100% of opportunities will be flagged. _Open: should the widget be retired and replaced by one that grades the canonical `Type` field, OR is the migration intentionally surfacing Type-less opportunities for cleanup?_
3. **`Missing Decision Reason` widget** (widget 1): the actual SF field name needs verification. The Phase 1 audit identified `Reason_Won_Lost__c` as the canonical reason field; this widget may grade a different (CFQ-related) field.
4. **Stage-specific aging thresholds** for widgets 10 (`pc_stage_age_within_threshold`) and 19 (`ph_stale_opportunities`): the spec proposes Stage 3 = 60d, Stage 4 = 45d, Stage 5 = 30d, Stage 6 = 15d as placeholders. These need Sales Ops sign-off before the audit grades against them.
5. **High-value threshold for widget 18** (`ph_high_value_stale_deals`): the existing report uses an implicit threshold. The spec needs an explicit number (e.g. EUR 1M). Verify in Phase 2 audit.
6. **Forecast accuracy PI list views** for widgets 11-14: the spec declares Pipeline Inspection native as canonical but does not pin the exact PI list view or metric config. Phase 2 source-contract work probes `PipelineInspectionListView` for active configurations and pins the right one per widget.
7. **CFQ field name** for widget 3 (`dq_missing_won_loss_cfq`): "CFQ" likely means "Customer Feedback Question" but the actual SF field needs sobject describe.
8. **Probability mismatch threshold** for widget 15 (`ph_probability_mismatch_by_stage`): the widget is marked "Under Construction" in the live dashboard. The spec needs the threshold definition. Treat as WIP in the audit.
